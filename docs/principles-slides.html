FAIR Data Principles

https://www.go-fair.org/fair-principles/

The FAIR principles were published in 2016 to improve the Findability, Accessibility, Interoperability, and Reusability of digital assets.

FAIR: Findable

  • F1. (Meta)data are assigned a globally unique and persistent identifier
  • F2. Data are described with rich metadata (defined by R1)
  • F3. Metadata clearly and explicitly include the identifier of the data they describe
  • F4. (Meta)data are registered or indexed in a searchable resource

Findability ensures that both humans and machines can discover data and metadata.

FAIR: Accessible

  • A1. (Meta)data are retrievable by their identifier using a standardised communications protocol
    • A1.1 The protocol is open, free, and universally implementable
    • A1.2 The protocol allows for an authentication and authorisation procedure, where necessary
  • A2. Metadata are accessible, even when the data are no longer available

Accessibility means data can be retrieved through standard protocols, even with proper authentication when needed.

FAIR: Interoperable

  • I1. (Meta)data use a formal, accessible, shared, and broadly applicable language for knowledge representation
  • I2. (Meta)data use vocabularies that follow FAIR principles
  • I3. (Meta)data include qualified references to other (meta)data

Interoperability enables data to work with applications and workflows for analysis, storage, and processing.

FAIR: Reusable

  • R1. (Meta)data are richly described with a plurality of accurate and relevant attributes
    • R1.1. (Meta)data are released with a clear and accessible data usage licence
    • R1.2. (Meta)data are associated with detailed provenance
    • R1.3. (Meta)data meet domain-relevant community standards

Reusability ensures data can be used in future research with clear licensing and provenance information.

CARE Principles

CARE Principles for Indigenous Data Governance

https://www.gida-global.org/care

CARE principles complement FAIR by addressing people and purpose in data governance, particularly for Indigenous data.

CARE: Collective Benefit

  • C1. For inclusive development and innovation
  • C2. For improved governance and citizen engagement
  • C3. For equitable outcomes

Data ecosystems should be designed to support Indigenous peoples’ rights and wellbeing.

CARE: Authority to Control

  • A1. Recognising rights and interests
  • A2. Data for governance
  • A3. Governance of data

Indigenous peoples have the right to govern data about their communities, territories, and resources.

CARE: Responsibility

  • R1. For positive relationships
  • R2. For expanding capability and capacity
  • R3. For Indigenous languages and worldviews

Those working with Indigenous data have responsibilities to share benefits and ensure respectful use.

CARE: Ethics

  • E1. For minimising harm and maximising benefit
  • E2. For justice
  • E3. For future use

Indigenous data should be used ethically to advance justice and support Indigenous innovation.

Operationalising CARE and FAIR

CARE and FAIR work together: FAIR focuses on data characteristics, CARE focuses on people and purpose.

Collections as Data

https://collectionsasdata.github.io/

Collections as Data encourages computational use of digitised and born-digital collections.

Collections as Data: Principles 1-6

  1. Collections as Data development aims to encourage computational use of collections
  2. Stewards are guided by ongoing ethical commitments
  3. Stewards aim to lower barriers to use
  4. Designed for everyone serves no one
  5. Shared documentation helps others find a path to doing the work
  6. Should be made openly accessible by default, except where ethical or legal obligations preclude it

The first six principles focus on access, ethics, and user needs.

Collections as Data: Principles 7-10

  1. Development values interoperability
  2. Stewards work transparently to develop trustworthy, long-lived collections
  3. Data as well as the data that describe those data are considered in scope
  4. Development is an ongoing process and does not necessarily conclude with a final version

The remaining principles emphasise interoperability, transparency, and iterative development.

Linked Data

The [World Wide Web] project merges the techniques of information retrieval and hypertext to make an easy but powerful global information system. The project started with the philosophy that much academic information should be freely available to anyone.

— Tim Berners-Lee, 1991

The Web was conceived with open access to information as a core principle.

Linked Data Principles

  1. Use URIs as names for things
  2. Use HTTP URIs so that people can look up those names
  3. When someone looks up a URI, provide useful information using standards (RDF, RDFS, SPARQL)
  4. Include links to other URIs so that they can discover more things

— Tim Berners-Lee, 2006

These four principles enable data to be connected and discoverable on the Web.

Resource Description Framework (RDF)

Everything in threes: subject, predicate, object \((s \ \vec{p} \ o)\)

RDF uses triples to create graphs of linked data with URIs for most components.

Linked Open Usable Data (LOUD)

LOUD extends LOD by emphasising usability

  • Coined by Robert Sanderson (2018, 2019)
  • Goal: Achieve the Semantic Web’s intent on a global scale in a usable fashion
  • Leverages community-driven and JSON-LD-based specifications
  • Five design principles for developer accessibility

LOUD makes linked data more accessible to software developers and end users.

LOUD Design Principles

  1. The right Abstraction for the audience
  2. Few Barriers to entry
  3. Comprehensible by introspection
  4. Documentation with working examples
  5. Few Exceptions, instead many consistent patterns

https://linked.art/loud/

These principles guide the development of usable linked data specifications.

LOUD Communities

Key Characteristics:

  • Synergy of social and technical integration
  • Shared expertise and leadership
  • Collaboration across boundaries
  • Inclusivity and diversity

Core Values:

  • Openness and friendliness
  • Transparency
  • Online and face-to-face meetings

LOUD communities like IIIF and Linked Art combine strong technical standards with welcoming social practices.

LOUD Standards

Three main specifications follow LOUD principles:

  1. International Image Interoperability Framework (IIIF)
  2. W3C Web Annotation Data Model
  3. Linked Art

All use JSON-LD and emphasise developer experience

These standards demonstrate how LOUD principles can be applied in practice.

IIIF Overview

  • A model for presenting and annotating content
  • A global community developing shared APIs
  • Implements software and exposes interoperable content

https://iiif.io

IIIF solves the problem of image delivery silos through standardised APIs.

IIIF: The Problem

Before IIIF:

  • Siloed image repositories
  • Duplication of efforts
  • Proprietary viewers
  • Limited interoperability

With IIIF:

  • Shared infrastructure
  • Reusable viewers
  • Interoperable resources
  • Global collaboration

IIIF eliminates the need for each institution to build custom image delivery systems.

IIIF Capabilities

  • Deep zoom with large images
  • Compare images
  • Reunify scattered collections
  • Search within
  • Storytelling
  • Crowdsourcing
  • Machine-generated annotations
  • Beyond images (A/V)

IIIF enables rich interactions with cultural heritage materials.

IIIF Specifications

Core APIs:

  • Image API - Delivers images with region, size, rotation parameters
  • Presentation API - Provides structural metadata for viewing

Additional APIs:

  • Authorization Flow, Change Discovery, Content Search, Content State

https://iiif.io/api

IIIF’s modular API design allows implementations to choose what they need.

IIIF Image API

URI Syntax:

{scheme}://{server}{/prefix}/{identifier}/{region}/{size}/{rotation}/{quality}.{format}

Example:

https://iiif.dasch.swiss/.../full/1000,/0/default.jpg

The Image API uses URI parameters to deliver images on demand.

IIIF Presentation API

Provides structural metadata using JSON-LD for:

  • Collections
  • Manifests
  • Canvases
  • Annotation Pages

The Presentation API describes how to display and navigate cultural heritage objects.

Web Annotation Data Model

Three-part structure:

  • Target: The resource being annotated
  • Body: The annotation content
  • Annotation: Links body and target

https://www.w3.org/TR/annotation-model/

The W3C Web Annotation model provides a standard way to associate content with resources.

Linked Art

A community and CIDOC Working Group defining:

  • Metadata application profile for cultural heritage
  • Technical means for convenient interaction (API)

https://linked.art

Linked Art applies LOUD principles to art museum data.

Linked Art Data Model

Level Technology
Conceptual Model CIDOC-CRM
Ontology RDF encoding of CRM 7.1 + extensions
Vocabulary Getty AAT, TGN, ULAN
Profile Object-based cultural heritage
API JSON-LD 1.1, REST

Linked Art builds on established cultural heritage standards with a focus on usability.

Linked Art Community

Major Institutions:

Linked Art has strong institutional support from leading museums and cultural organisations.

Linked Art Overview

Linked Art models the full lifecycle of cultural heritage objects and related entities.

IIIF and Linked Art Integration

Complementary roles:

  • IIIF for digital access
  • Linked Art for semantic description

IIIF and Linked Art work together to provide both access and rich metadata.

LOUD Infrastructure Example

LOUD standards enable sophisticated infrastructure for cultural heritage access and research.

LOUD in a Nutshell

  • Community-driven: Grassroots development with transparency
  • Implementation-focused: Specifications developed alongside working software
  • Semantic interoperability: Even at the cost of ontological purity
  • Common denominators: For institutions, public bodies, and research projects

LOUD practices enhance semantic interoperability across cultural heritage

LOUD represents a pragmatic approach to linked data that prioritises usability and adoption.
