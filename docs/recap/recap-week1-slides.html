What is Open Data?

Definitions

  • Openness: freely access, use, modify, and share ‚Äî for any purpose
  • Legally open: under an open licence permitting reuse and redistribution
  • Technically open: machine-readable, no more than reproduction cost
  • Metadata: data about data ‚Äî no fixed boundary between the two

A Brief History

  • 1942 ‚Äî Merton: researchers must contribute to the ‚Äúcommon pot‚Äù
  • 1995 ‚Äî term ‚ÄúOpen Data‚Äù first appears (geophysical/environmental data)
  • 2005 ‚Äî Open Knowledge Foundation publishes the Open Definition
  • 2007 ‚Äî Sebastopol Meeting: 8 principles of open government data
  • 2009 ‚Äî Berners-Lee at TED: ‚ÄúRaw Data Now‚Äù

Open data requires both legal and technical openness. The concept predates the term ‚Äî the idea that publicly funded knowledge must benefit the public has roots in 1940s science.

Movements and Principles

Movements

  • Open Access (OA): free, online access to scholarly publications ‚Äî Gold, Green, Diamond, Hybrid, Bronze, Blue, Black
  • Open Science / Open Scholarship: entire research lifecycle made open ‚Äî data, methods, peer review, software
  • FLOSS: Free/Libre and Open Source Software ‚Äî 4 freedoms (run, study, change, distribute)

Principles

  • FAIR: Findable, Accessible, Interoperable, Reusable ‚Äî the technical framework for data sharing
  • CARE: Collective Benefit, Authority to Control, Responsibility, Ethics ‚Äî governance for Indigenous data
  • Collections as Data: GLAM collections reimagined as computational resources ‚Äî openness by default, interoperability, ethical stewardship
  • LOUD: Linked Open Usable Data ‚Äî developer-friendly, JSON-LD, community-driven

FAIR tells us how to structure data. CARE tells us whose interests must be protected.

These movements and principles are complementary, not competing. Open Access focuses on publications, Open Science on the full research process, FLOSS on tools. FAIR and CARE together ensure data is both technically sound and ethically grounded.

From Tables to Triples

A museum wants to describe Van Gogh‚Äôs Starry Night. Two approaches:

Relational Database

Three siloed tables, linked by local integer IDs:

Table Key columns
Artist id, name, nationality, birthday
Artwork id, title, date, material, artist_id ‚Üó
Museum id, name, location
  • Relationships via foreign keys + SQL JOIN
  • IDs are local ‚Äî id: 1 means nothing outside this database
  • Schema is rigid ‚Äî adding a column requires a migration
  • Data is siloed ‚Äî not designed to link to the Rijksmuseum‚Äôs database

Triplestore

Nine triples, every relationship is explicit data:

Subject Predicate Object
ex:vangogh rdf:type foaf:Person
ex:vangogh foaf:name "Vincent van Gogh"
ex:vangogh schema:nationality "Dutch"
ex:vangogh ex:created ex:starrynight
ex:starrynight rdf:type schema:Painting
ex:starrynight schema:name "The Starry Night"
ex:starrynight schema:location ex:moma
ex:moma rdf:type schema:Museum
ex:moma foaf:name "Museum of Modern Art"
  • IDs are global URIs ‚Äî wd:Q5582 is Van Gogh everywhere
  • Schema is flexible ‚Äî add triples without breaking anything
  • Relationships are globally linked ‚Äî any entity can reference any URI

This is the core Linked Data shift. In a relational DB, relationships are implicit (foreign keys, JOIN operations) and local. In a triplestore, every relationship is an explicit triple with a globally meaningful URI. The same Van Gogh (wd:Q5582) can be referenced by the Getty, Wikidata, and the Rijksmuseum simultaneously.

The Same Data as a Knowledge Graph

In this graph: warm orange rounded nodes are named resources (URIs); dark red rectangles are class types (rdf:type targets); dark teal rectangles are literal string values. The edge labels show the predicate ‚Äî each label has a light background to prevent overlap with arrows. ex:moma could link to wd:Q188740 (MoMA on Wikidata), which in turn links to hundreds of other artists and artworks ‚Äî that is the Web of Data. The graph is open-ended ‚Äî any node can link out to any other URI on the web, connecting data across institutions globally.

Persistent Identifiers (PIDs)

Why PIDs matter: Without globally unique, stable identifiers, data remains siloed. PIDs ensure the same entity is recognised everywhere.

PID Characteristics

  • Unique: no two entities share the same PID
  • Persistent: remains valid even if hosting infrastructure changes
  • Resolvable: points to current location via standard resolver
  • Interoperable: recognised across systems and communities
  • Human & machine-readable: accessible both visually and programmatically

PIDs are the infrastructure that makes the Web of Data work. Without them, you can‚Äôt reliably link across institutions. A good PID must be both technically resolvable and institutionally committed to long-term maintenance.

Common PID Types

PID Type Identifies Example Resolver
DOI Publications, datasets, objects 10.1038/nature12373 https://doi.org/10.1038/nature12373
ORCID Researchers, scholars 0000-0002-4711-5759 https://orcid.org/0000-0002-4711-5759
ROR Research organisations 01xkakk17 https://ror.org/01xkakk17
ARK Cultural heritage objects ark:12148/btv1b108473193/ https://n2t.net/ark:12148/btv1b108473193/
Handle General-purpose resources 20.500.14716/127190 https://hdl.handle.net/20.500.14716/127190

DOI is the industry standard for academic publishing, ORCID for researchers, ROR for institutions, and ARK for cultural heritage objects. Each PID type has a dedicated resolver: doi.org, orcid.org, ror.org, hdl, and n2t.net (ARK).

Key Takeaways

  • Open Data = legally open (licence) + technically open (machine-readable format)
  • OA models differ in who pays, when access is granted, and whether a licence exists ‚Äî Bronze is free to read but has no licence
  • FAIR structures data for machines; CARE protects people‚Äôs interests; Collections as Data and LOUD make it usable
  • Linked Data replaces local integer IDs with global URIs and foreign keys with explicit triples
  • A triplestore stores subject ‚Üí predicate ‚Üí object triples, queried via SPARQL ‚Äî not SQL
  • Persistent Identifiers (PIDs) ensure entities are recognised globally and unambiguously across the Web of Data
  • The Web of Data connects the same entity (e.g., Van Gogh) across museums, archives, and knowledge bases worldwide

Open data is not just about making files available ‚Äî it is about creating an interconnected web of usable, ethically governed knowledge.

These seven points capture the conceptual arc of Week 1: from the definition of open data and its history, through movements and principles, to the technical foundations of Linked Data, PIDs, and the global web they enable together.
